## Context

`@astrojs/starlight@0.41.3` requires `astro@^7.0.2`. The site was on 6.4.8.
Starlight `0.40.x` still supports `astro@^6.4.5`, so pinning an older Starlight
was a real alternative that avoided touching the framework.

## Decision

Upgrade to Astro 7.1.3 and take the latest Starlight, rather than pin.

## Consequences

- **Markdown pipeline changed.** Astro 7 defaults to Sätteri instead of
  remark/rehype. `rehype-external-links` only kept working because
  `@astrojs/markdown-remark` was added and `markdown.rehypePlugins` was migrated
  to `markdown.processor = unified({…})`.
- **Rust compiler is now the only compiler.** It errors on unclosed non-void
  elements — one self-closing `<script />` in `blog/[post].astro` had to be
  fixed.
- **Vite 8 / rolldown** introduced a native-binding dependency, which surfaced a
  local-only problem: an x64 Node under Rosetta can't load the arm64 binding.
  See [mac setup](/docs/dev/mac-setup/). CI was unaffected.
- Everything else — the local `spectre` integration, `astro/zod`, view
  transitions — carried over untouched.

**Why not pin:** pinning would have parked the site one major version behind and
made the same upgrade unavoidable later, with more accumulated drift. The
upgrade was done as its own commit so it could be reverted independently of the
notes feature.
